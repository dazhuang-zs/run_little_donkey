# 计算机毕业生面试技巧与常见问题生存指南

> 本指南旨在帮助计算机专业毕业生系统掌握从面试流程到技术通关的全套方法论。无论你是985、211还是普通本科，目标只有一个：把面试官问到的东西都答出来，把答出来的东西都讲清楚。学历是门票，实力才是通关证。

---

## 第一部分：面试流程与形式

### 1.1 技术面试的完整流程

互联网校招的标准流程通常如下：

**流程 A：大厂标准路径**
```
网申/内推 → 笔试（算法题）→ 一面（技术基础）→ 二面（技术深度/项目）→ 三面（架构/系统设计）→ HR面 → 录用
```

**流程 B：中小厂/创业公司路径**
```
简历筛选 → 一面（技术面，侧重手撕代码）→ 二面（技术负责人，侧重项目与架构）→ HR面 → 录用
```

**流程 C：外企路径**
```
网申/OA（在线测评）→ 电面（电话技术面）→ Onsite（多轮现场/视频面，通常3-5轮）→ HR谈判
```

> **普通院校同学注意**：流程中的轮数不是固定不变的。大厂有时候一面挂直接结束，有时候加面是积极信号（说明前面的面官给了positive feedback想多了解你）。不要过度解读轮数变化，把精力放在每轮的表现上。

### 1.2 各轮面试的考察重点与应对策略

#### 一面：基础功筛查（45-60分钟）

**考察重点**：计算机基础 + 算法编码能力

典型题目分配：
- 自我介绍（2-3分钟）
- 计算机基础提问（15-20分钟）：操作系统、网络、数据结构
- 手撕代码（20-30分钟）：1-2道算法题，难度以LeetCode简单和中等为主
- 反问环节（3-5分钟）

**应对策略**：

```markdown
✅ 一面通过特征：
- 代码能跑，bug少，思路清晰
- 基础问题答对80%以上
- 不会的题能展示思考过程，不乱猜

❌ 一面挂的典型原因：
- 代码写不出来且没有思路
- 基础概念明显错误（如"进程是线程的子集"）
- 沟通表达能力差，问东答西
- 10分钟还没理解题意
```

**话术模板**：

> 面试官："先做个自我介绍吧。"
>
> "我叫xxx，毕业于xxx大学计算机专业。在校期间主要做了两个项目：一个是……（用2分钟说清楚技术栈、你的角色、核心成果）。我对后端开发比较感兴趣，暑假在xxx公司实习了两个月，主要做……（一句话总结）。今天应聘的是xxx岗位，希望能深入了解贵司的技术方向。"

---

#### 二面：技术深度与项目挖掘（45-60分钟）

**考察重点**：项目经验深度 + 技术视野 + 问题分析能力

典型题目：
- 项目深挖（20-30分钟）：技术选型、难点、解决方案、效果
- 系统设计基础题（10-15分钟）：设计一个短链接系统、设计一个秒杀系统
- 场景分析（10-15分钟）：如果Redis挂了怎么办、线上服务响应慢如何排查

**应对策略**：

二面是你展示"不只是会做题"的最佳机会。面试官往往级别较高，关心你**如何思考**，而不是你**背了多少答案**。

> **普通院校同学注意**：没有大厂实习经历不是世界末日。二面可以讲课程项目、毕业设计、开源贡献（哪怕是给知名项目提了文档PR）、竞赛获奖项目。关键是讲清楚"你解决了什么问题，用什么方法，达到了什么效果"。

---

#### 三面/交叉面：架构视野与综合能力（30-45分钟）

**考察重点**：系统设计能力 + 技术判断力 + 文化匹配度

典型题目：
- 设计一个微博 feed 系统
- 如何设计一个高可用的分布式架构
- 你对某项技术趋势的看法
- 团队协作中的冲突如何处理

**应对策略**：

这一轮更多考察的是你的思考框架。答案本身没有标准答案，面试官在观察你如何拆解问题、如何在约束条件下做trade-off。

---

#### HR面：综合素养与稳定性（20-30分钟）

**考察重点**：价值观匹配度、稳定性、沟通表达能力、薪资期望

典型问题：
- 为什么选择我们公司/这个岗位？
- 你最有成就感的一件事是什么？
- 你的职业规划是什么？
- 能介绍下你的优点和缺点吗？

*详见第三部分专项讲解。*

---

### 1.3 远程面试 vs 现场面试

#### 远程视频面试注意事项

**硬件与网络**：
- 提前测试网络，推荐使用有线网络连接
- 笔记本电量充满，插着电源
- 找一个安静、光线充足、背景整洁的房间
- 耳机比外放更安全，避免回声

**环境准备**：
- 提前关闭微信、QQ等即时通讯软件的通知
- 手机静音或开飞行模式
- 备选方案：手机热点（网络不稳定时备用）

**代码环节**：
- 大厂常用平台：CodeSignal、Hackerrank、牛客网
- 提前在对应平台做一次模拟练习，熟悉输入输出格式
- 手撕代码时边写边说，让面试官看到你的思考过程

**眼神与姿态**：
- 看摄像头而非屏幕，这样面试官感觉你在看他
- 坐姿端正，不要葛优瘫
- 适度点头和回应，展现你在倾听

> **话术**：远程面试中，如果网络卡顿，主动说"不好意思，网络有点不稳定，请等我一下"，不要假装没卡顿导致双方都听不清。

#### 现场面试注意事项

**必须带的东西**：
- 个人简历（多打印5份）
- 本子、笔
- 身份证、学生证（入场凭证）
- 手机（保持静音）

**前一天晚上**：
- 确认路线，计算好通勤时间，提前10分钟到场
- 穿干净整洁的衣服（不一定正装，互联网公司smart casual即可）
- 早点休息，不要熬夜

**面试现场**：
- 进入会议室后先向面试官问好，握手要适度
- 代码写在白板上字迹要清晰，边写边解释
- 遇到不会的题，征询"能否给点提示"，比干坐着好
- 结束后感谢面试官，礼貌道别

## 第二部分：技术面试技巧

### 2.1 手撕代码环节：如何沟通思路、处理边界条件、优化解法

手撕代码是技术面试的核心环节，也是最容易紧张的部分。很多人不是不会做题，而是死在了"想清楚再写"和"写出来再解释"之间的节奏失调上。

#### 标准四步法

**第一步：clarify（5分钟以内）**

> 拿到题目后，不要立刻动手写。先问清楚边界条件和需求。

常见clarify问题：
- "数据规模是多少？时间和空间复杂度有要求吗？"
- "输入的数据是否有序？是否包含重复元素？"
- "返回的是什么？索引还是具体的值？"
- "是否需要处理空输入、非法输入的情况？"

**第二步：brute force → optimize（5-10分钟）**

不要一上来就追求最优解。标准策略是：

1. 先想出一个暴力解（即使复杂度差也要能写出来）
2. 解释暴力解的思路和复杂度
3. 在面试官引导下或自己想到优化方向
4. 写出优化解

```markdown
话术示例：
"我先想一个最简单的做法：用两层循环遍历所有子数组，时间复杂度O(n²)，空间O(1)。
这个做法对于小规模数据可以，但n很大时性能不行。
我注意到这个问题有最优子结构，可以考虑用前缀和的思想优化到O(n)……"
```

**第三步：写代码（边写边说）**

- 写代码时不要完全沉默，说出你在做什么
- 代码写完后，主动检查：数组越界、空指针、边界条件
- 手写代码时变量名起得有意义，不要用a、b、c糊弄

**第四步：test（自测）**

```markdown
手动跑一遍测试用例：
输入: [2, 7, 11, 15], target = 9
输出: [0, 1]（解释：nums[0] + nums[1] = 2 + 7 = 9）

边界case:
- 空数组 → 返回[]
- 单元素 → 返回[]
- target为负数 → 正常处理
```

#### 常见边界条件清单

| 场景 | 必须考虑的边界 |
|------|--------------|
| 数组相关 | 空数组、单个元素、所有元素相同 |
| 字符串 | 空字符串、纯空格、特殊字符、大小写 |
| 链表 | 空链表、单节点、环形链表 |
| 二叉树 | 空树、单节点、完全不平衡 |
| 数字运算 | 溢出（int vs long）、负数、零 |

#### 普通院校同学建议

如果算法基础一般，建议按以下优先级准备：

```
必刷熟练（高频出现）：
- 双指针（快慢指针）
- 哈希表（空间换时间）
- 排序算法（快速排序、归并排序）
- 二分查找及其变体
- 递归与回溯（理解思想，会写模板）

高频但不必死磕（理解套路，面试能想出来即可）：
- 动态规划（背包、LIS、LCS）
- BFS/DFS（图相关）
- 堆（topK问题）
- 位运算
```

**LeetCode不用刷完**。把HOT 100刷两遍，理解每道题背后的思路，比刷500道囫囵吞枣强十倍。

### 2.2 项目深挖：如何准备项目讲解、应对连环追问

项目深挖是二面/三面的重头戏。面试官不是在考你项目是什么，而是在通过项目看你**解决问题的能力**。

#### 项目讲解：STAR法则 + 技术深度

```markdown
STAR法则：
- S (Situation): 项目背景是什么
- T (Task): 你负责什么/要解决什么问题
- A (Action): 你具体做了什么（重点）
- R (Result): 最终效果如何（量化）
```

**示例项目讲解（30秒版本）**：

> "这个项目是一个校园二手交易平台（S），我负责后端用户模块和商品搜索模块的开发（T）。我调研了三种搜索方案，最终选择Elasticsearch方案，通过优化索引结构和分词器，使搜索响应时间从500ms降到80ms，上线后日均搜索量超过2万次（R）。"

**示例项目讲解（5分钟版本，适合深挖）**：

> "这个项目最大的难点是并发场景下的库存超卖问题。最初我们用数据库乐观锁实现，但压测发现QPS只能到800左右。后来我查阅资料，改用Redis + Lua脚本的原子操作方案，QPS提升到5000以上……"

#### 应对连环追问的策略

**追问类型 1：技术选型类**
> "为什么选择Redis而不是本地缓存？"
> 
> 回答思路：先说结论，再说为什么，然后说如果换方案会怎样。
> 
> "因为项目需要支持分布式部署，本地缓存无法跨节点共享。Redis提供天然的分布式支持，读写性能也很高。如果是单机应用，本地缓存会更轻量，不需要额外部署Redis服务。"

**追问类型 2：深度追问类**
> "那你这个Redis分布式锁具体是怎么实现的？"
> 
> 遇到这种问题，说明你踩到了面试官的舒适区（他的专业领域），是展示深度的机会。提前准备项目中每个技术点的底层原理。

**追问类型 3：失败/问题类**
> "这个项目有遇到过什么问题或者失败的地方吗？"
> 
> 这个问题几乎是必问的。提前准备：
> 
> "遇到过。上线第一周商品图片上传失败率很高（10%左右）。排查后发现是Nginx的client_max_body_size配置过小，加上异步上传时文件正在写入就返回了。后来增加了上传状态确认机制并调整了Nginx配置，故障率降到0.1%以下。"

#### 普通院校同学的项目准备建议

没有大厂实习经历的，可以从以下角度挖掘项目：

1. **课程设计做到极致**：把课设项目的技术细节搞清楚，不要只停留在"能跑"
2. **毕设项目**：这是你最有深度的项目，务必对每个技术决策都准备"为什么"
3. **GitHub开源参与**：哪怕只是修复bug、改进文档，也是真实项目经验
4. **竞赛项目**：ACM、数学建模、挑战杯等比赛项目
5. **自己做的有趣项目**：比如写了个爬虫、写了个博客系统，写到简历上也可以，但必须能讲出技术细节

**关键**：不要把项目写成技术堆砌清单。简历上写一个项目，脑子里要能讲20分钟。

### 2.3 系统设计/场景题：应对策略与思考框架

系统设计题没有标准答案，面试官关心的是你如何思考。

#### 五步思考框架

**Step 1：理解需求 & 边界确认**
- 需要支持多少用户？DAU多少？
- 核心功能是什么？优先级排序？
- 需要支持哪些读写模式？

**Step 2：高层设计**
- 估算需要多少服务器、存储容量
- 画出核心组件（API层、存储层、缓存层、消息队列）

**Step 3：核心组件设计**
- 数据库选型（MySQL / MongoDB / Cassandra）
- 缓存策略（Cache-Aside / Read-Through）
- 异步处理（消息队列解耦）

**Step 4：扩展性考虑**
- 如何支持数据量增长（分库分表）
- 如何保证高可用（主备、多活）
- 如何做负载均衡

**Step 5：权衡取舍**
- 一致性 vs 可用性（CAP理论的实际应用）
- 开发复杂度 vs 性能收益

#### 常见系统设计题目模板

**设计一个短链接系统**：

```markdown
需求澄清：
- 每日生成量级？10万/天 → 2亿条/天 → 存储方案不同
- 链接有效期？永久还是临时？
- 是否需要统计点击量？

存储选型：
- 7位短码 + 原始URL → 存储在MySQL（url列建索引）
- 短码生成：发号器（自增ID转62进制）vs 哈希（需处理碰撞）

性能优化：
- Redis缓存热数据（访问频率Top N的短链）
- CDN加速静态资源
- 读写分离应对读多写少

高可用：
- 主备MySQL + Redis Cluster
- 限流防刷（同一IP每秒请求次数限制）
```

**设计一个秒杀系统**：

```markdown
核心挑战：瞬时高并发 → 库存超卖

解决方案：
1. 前端：验证码防刷 + 按钮置灰
2. 接入层：Nginx限流（令牌桶算法）
3. 业务层：Redis预减库存 + 消息队列异步下单
4. 库存操作：Redis Lua原子扣减，数据库最终一致性
5. 防超卖：下单前双重校验（Redis扣减成功 + 数据库库存>0）
```

> **面试话术**：遇到系统设计题，可以主动说"我先估算一下数据规模和性能需求"，这展示你有工程思维，不是纸上谈兵。

### 2.4 不会的问题如何处理

这是面试中最考验心态的环节。处理不好直接挂，处理好了反而能加分。

#### 分级处理策略

**第一级：完全不会，没思路**

> 话术："这个知识点我之前了解不深，我尝试从已知知识推导一下……"
> （尝试类比、分解、画图）
> 如果推不出来，直接说："这个问题我现在没有很清晰的思路，面试结束后我会去深入学习这个方向。能给我简单介绍一下吗？"

**第二级：有印象，但记不清细节**

> 话术："这个概念我了解过，核心原理是……（说大概），具体实现细节我记不太清了，面试后我去补充一下。"
> 
> 切忌：不要乱编、瞎猜、胡扯。面试官问这个问题说明这是他们的专业领域，你乱说反而暴露水平。

**第三级：有思路但不完整**

> 话术："这个问题我从两个角度来分析……（说出你确定的部分）。不过还有几个方面我不太确定，比如……，您能帮我补充一下吗？"
> 
> 主动承认边界 + 展示学习意愿，比硬撑强得多。

#### 绝对禁止的行为

- ❌ 沉默不语（超过1分钟）
- ❌ 假装在思考实际在发呆
- ❌ 乱编乱造（面试官稍一追问就露馅）
- ❌ 转移话题到别的知识点（容易被认为在逃避）
- ❌ 抱怨问题太难（负能量暴露）

## 第三部分：HR面试与谈薪

### 3.1 HR面的真实目的与常见陷阱

HR面的本质是**筛选风险**：你是不是正常人？你会不会入职后很快离职？你要多少钱？

#### 常见问题与回答策略

**问题1："为什么从上家公司/实习离职？"**

> 陷阱：抱怨前公司/前领导。
> 
> ✅ 正确姿势：正向表述，聚焦成长。
> "上家公司给了我很好的成长机会，但我的职业规划更倾向于做xxx方向，而那边主要是xxx方向，所以希望寻找更匹配的平台。"

**问题2："你最有成就感的一件事是什么？"**

> 陷阱：说一些太普通的事（"考试考得不错"），或太ego的事（"我一个人完成了整个项目"）。
> 
> ✅ 正确姿势：STAR法则，具体量化，团队贡献。
> 参考项目深挖的STAR模板，体现你的能力和成长。

**问题3："你的职业规划是什么？"**

> 陷阱：说"三年当主管"（太功利），说"不知道"（没规划），说"创业"（稳定性差）。
> 
> ✅ 正确姿势：技术成长路线 + 业务价值 + 长期愿景。
> "我希望1-2年在技术上深耕，成为某个领域（如分布式系统/后端架构）的专家，3-5年能够独立负责核心模块，为团队带来技术价值。"

**问题4："你还有什么问题问我？"**

> 这是送分题，但很多人浪费了。
> 
> ✅ 值得问的问题：
> - "团队的技术栈是什么？入职后有mentor带吗？"
> - "这个岗位的绩效考核维度是怎样的？"
> - "部门的业务发展方向是什么？"
> - "入职后3-6个月的主要工作是什么？"
> 
> ❌ 不要问的问题：
> - "加班多吗"（除非真的很在意）
> - "能不能解决户口"（除非这是唯一标准）
> - "有什么福利"（入职前不是重点）

### 3.2 期望薪资怎么说

#### 市场调研方法

1. **看准网、牛客网offer区**：查目标公司+目标岗位的校招薪资范围
2. **脉脉匿名区**：可以参考，但数据不一定准确
3. **同届同学交流**：互相分享offer信息（不要互相藏着掖着，信息对称对双方都有利）
4. **Offershow小程序**：应届生薪资数据集中地

#### 报价策略

**原则：先了解市场，再报出自己的数字**

```
情况1：手里有竞争offer
→ 用竞争offer倒逼："基于我的调研和手上其他offer，这个岗位的市场价在xxk左右，我希望是xxk"

情况2：手里没有offer
→ 先试探："我更看重平台和成长机会，想先了解一下贵司的薪资结构是怎样的？"
→ 如果HR逼问，给一个区间而非具体数字："考虑到我的背景和市场行情，我希望在xx-xxk之间"

情况3：普通院校/无实习经历，底气不足
→ 不要报太低，市场价是多少就报多少。报低了入职后吃亏，且不利于后续涨薪
→ 可以加一句："当然，如果平台发展前景好，薪资方面可以有一定弹性"
```

> **重要**：第一轮报价往往会决定后续的谈判空间。报高了显得不切实际，报低了自己吃亏。建议给自己设定一个**底线薪资**（低于这个绝对不去），在此基础上留出5-10%的谈判余地。

#### 薪资构成要看清楚

HR报给你的数字，要问清楚是：
- 总包还是base？（股票、奖金、签字费都要算进去）
- 试用期和转正后薪资一样吗？（通常一样）
- 五险一金按什么基数缴纳？（实际工资 vs 最低工资，差距很大）
- 年终奖几个月？（通常2-6个月，看公司）

### 3.3 Offer选择与比较维度

拿到多个offer后，从以下维度打分：

| 维度 | 权重建议 | 评估问题 |
|------|---------|---------|
| 薪资待遇 | 25% | 三年总包是多少？涨幅预期？ |
| 技术成长 | 25% | 技术栈是否前沿？有技术挑战吗？ |
| 业务前景 | 20% | 公司/部门盈利吗？赛道前景如何？ |
| 团队氛围 | 15% | 直属领导风格？团队规模？ |
| 城市/通勤 | 10% | 房价、生活成本、通勤时间 |
| 加班文化 | 5% | WLB（工作生活平衡）对你重要吗？ |

> **普通院校同学建议**：第一份工作，成长权重 > 薪资权重。5000的差距三年后回头看微不足道，但技术能力的差距会决定你五年后的选择空间。

### 3.4 三方协议与违约注意事项

三方协议（就业协议书）是学校、公司、毕业生三方签订的正式协议，有法律效力。

**签约前必读**：
- 违约金是多少？（通常5000-20000元，个别大厂违约金很高）
- 毁约流程是什么？学校什么时候可以申请新的三方？
- 毁约后新三方什么时候能拿到？

**毁约注意事项**：
1. 拿到新offer后再毁约，不要裸签
2. 毁约前确认新公司能等你（有些公司审批流程很长）
3. 和原公司HR沟通时态度要好，有些公司毁约很简单
4. 毁约成本（违约金 + 时间成本）要算进offer比较里

**特殊情况**：
- 试用期和实习期不是一回事。实习期通常没有三方，毕业后才正式签劳动合同
- 违约金条款在法律上有上限限制，但实践中仍需履行
- 如果遇到公司毁约（虽然少见），可以要求赔偿

### 4.1 Java高频

#### JVM ⭐⭐⭐

```markdown
1. JVM组成：堆（Heap）、方法区、虚拟机栈、本地方法栈、程序计数器、GC
   → 堆是GC重点区域，分为新生代（Eden+Survivor）和老年代

2. 垃圾回收算法：
   - 标记-清除（产生碎片）
   - 复制（适合新生代，无碎片但浪费空间）
   - 标记-整理（适合老年代，无碎片）
   → 新生代用复制，老年代用标记-整理

3. 常见GC器：
   - Serial（单线程STOP THE WORLD）
   - Parallel Throughput（多线程，关注吞吐量）
   - CMS（并发标记清除，低停顿但有碎片）
   - G1（分区算法，可设停顿时间目标）
   - ZGC/Shenandoah（低延迟，JDK11+）

4. 类加载机制：
   加载→验证→准备→解析→初始化→使用→卸载
   双亲委派：避免类的重复加载，保证Java核心类库安全

5. 常见调优参数：
   -Xms/-Xmx（堆大小）、-Xmn（新生代）、-XX:+UseG1GC
```

#### 并发编程 ⭐⭐⭐

```markdown
1. 线程实现方式：
   - 继承Thread类
   - 实现Runnable接口（更常用）
   - 实现Callable + FutureTask（可以有返回值）
   - 线程池（ExecutorService，最推荐）

2. 线程池参数：
   corePoolSize、maxPoolSize、keepAliveTime、workQueue、threadFactory、handler
   → 拒绝策略：CallerRunsPolicy、AbortPolicy、DiscardPolicy、DiscardOldestPolicy

3. synchronized vs Lock：
   - synchronized：JVM内置，自动释放，可重入
   - Lock：需手动unlock，可tryLock，支持公平/非公平
   → 优先用synchronized，复杂场景用Lock

4. volatile：
   保证可见性 + 有序性（禁止指令重排），不保证原子性
   → 原理：写后刷主存，读从主存取

5. CAS（Compare And Swap）：
   乐观锁思想，AtomicInteger底层用的就是这个
   → ABA问题：加版本号解决（AtomicStampedReference）

6. AQS（AbstractQueuedSynchronizer）：
   ReentrantLock、CountDownLatch、Semaphore的底层实现
   → 核心：state变量 + CLH队列

7. 线程间通信：
   - volatile/synchronized
   - wait/notify
   - Condition
   - 消息队列
```

#### 集合框架 ⭐⭐⭐

```markdown
List：
- ArrayList：数组实现，尾插O(1)，中间插入O(n)
- LinkedList：双向链表，头尾操作O(1)
- Vector：线程安全，synchronized，效率低

Map：
- HashMap：数组+链表（JDK1.8+红黑树）⭐⭐⭐
  → 扩容机制、负载因子0.75、链表转红黑树条件（长度≥8且数组≥64）
- ConcurrentHashMap：分段锁（JDK1.7）→ CAS+synchronized（JDK1.8）
- TreeMap：红黑树实现，有序
- LinkedHashMap：插入顺序/访问顺序

Set：
- HashSet：底层是HashMap（value用固定对象）
- TreeSet：底层是TreeMap

Queue：
- LinkedList（双向队列）
- PriorityQueue（堆，小顶堆）
- Deque（双端队列）
- BlockingQueue（ArrayBlockingQueue、LinkedBlockingQueue）
```

#### Spring ⭐⭐

```markdown
1. IoC/DI：
   控制反转，将对象创建权交给Spring容器
   DI（依赖注入）三种方式：构造器注入、setter注入、字段注入

2. AOP：
   面向切面编程，通知（Before/After/Around）+ 切点（Pointcut）
   应用场景：事务管理、日志、权限校验

3. Spring事务传播行为：

4. Spring Boot自动装配：
   @SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan

5. Spring Bean生命周期：
```

### 4.2 数据库高频

#### 索引 ⭐⭐⭐

```markdown
1. 索引类型：
   - 主键索引（PRIMARY）：唯一且非空，一表一个
   - 唯一索引（UNIQUE）：值唯一
   - 普通索引（INDEX）：加速查询
   - 全文索引（FULLTEXT）：文本搜索

2. B+树 vs B树：
   - B+树非叶子节点不存储数据，只存储索引 → 同样高度能存更多索引
   - B+树叶子节点链表相连 → 范围查询友好
   - B树所有节点都存数据 → 查询不稳定

3. 最左前缀原则：
   复合索引(a,b,c)，查询能用上索引：a / a,b / a,b,c
   不能用：b / c / b,c

4. 索引失效情况：
   - 使用函数/计算（WHERE YEAR(created_at) = 2024）
   - 类型转换（字符串字段用数字查询）
   - LIKE以%开头
   - OR前后有一个不带索引
   - NOT IN / NOT EXISTS

5. 聚集索引 vs 非聚集索引：
   聚集索引：数据行的物理顺序与索引顺序一致（InnoDB主键索引）
   非聚集索引：数据行与索引分开存储
```

#### 事务 ⭐⭐⭐

```markdown
ACID：
- Atomicity（原子性）：事务是最小执行单位
- Consistency（一致性）：事务执行前后数据状态一致
- Isolation（隔离性）：并发事务互不干扰
- Durability（持久性）：事务提交后数据永久保存

隔离级别（从低到高，并发问题从多到少）：
1. Read Uncommitted：脏读、不可重复读、幻读
2. Read Committed：不可重复读、幻读（Oracle默认）
3. Repeatable Read：幻读（MySQL默认，MVCC解决大部分）
4. Serializable：完全串行，性能差

MVCC（Multi-Version Concurrency Control）：
- 隐藏列：trx_id（事务ID）+ roll_pointer（指向undo日志）
- ReadView：m_ids（活跃事务列表）、min_trx_id、max_trx_id
- 读已提交：每次快照读都生成新ReadView
- 可重复读：同一个事务用同一个ReadView
```

#### 锁 ⭐⭐

```markdown
1. 锁类型：
   - 共享锁（S锁）：SELECT ... LOCK IN SHARE MODE
   - 排他锁（X锁）：SELECT ... FOR UPDATE / INSERT / UPDATE / DELETE

2. 行锁 vs 表锁：
   - InnoDB行锁：锁索引，非主键索引会锁主键
   - 表锁：表锁住全表（无索引或锁冲突时升级）

3. 死锁：
   - 产生条件：互斥、占有并等待、不可抢占、循环等待
   - 解决：MySQL会自动回滚小事务；设计时按固定顺序访问资源

4. Next-Key Lock：
   InnoDB RR级别下，锁住记录本身+前后区间
   → 防止幻读（Record Lock + Gap Lock）

5. 乐观锁 vs 悲观锁：
   - 悲观锁：假设冲突，select for update
   - 乐观锁：假设无冲突，更新时检查version字段
```

#### SQL优化 ⭐⭐

```markdown
1. EXPLAIN分析：
   - type: const > eq_ref > ref > range > index > ALL（尽量ref以内）
   - key: 实际使用的索引
   - rows: 扫描行数估算
   - Extra: Using filesort、Using temporary（需优化）

2. 优化技巧：
   - 分页优化：延迟关联（子查询先定位ID，再JOIN）
   - JOIN优化：小表驱动大表（IN vs EXISTS）
   - 避免SELECT *，只查需要的列
   - 批量操作代替循环单条

3. 分库分表：
   垂直分：按业务拆分（减少宽度）
   水平分：按数据拆分（减少行数）
   中间件：ShardingSphere、MyCat

4. 主从复制：
   binlog（Statement/Row/Mixed）→ 从库IO线程拉取 → SQL线程执行
   读写分离：写主库，读从库（延迟问题需处理）
```

### 4.3 网络高频

#### HTTP ⭐⭐⭐

```markdown
1. HTTP方法：
   GET：获取资源（幂等、安全）
   POST：提交数据（不幂等）
   PUT：替换资源（幂等）
   DELETE：删除资源（幂等）
   HEAD：获取头部（无body）
   OPTIONS：跨域预检

2. GET vs POST：
   - GET参数在URL，有长度限制（浏览器约2KB），可缓存
   - POST参数在body，大小无限制，不可缓存
   - 两者都明文传输，HTTPS才加密

3. HTTP状态码：
   2xx: 200成功、201创建、204无内容
   3xx: 301永久重定向、302临时重定向、304缓存未修改
   4xx: 400参数错误、401未认证、403禁止访问、404未找到
   5xx: 500服务器错误、502网关错误、503服务不可用、504超时

4. HTTP/1.0 vs 1.1 vs 2.0 vs 3.0：
   1.1：长连接（Connection: keep-alive）、管道化、带宽优化
   2.0：多路复用（一个TCP连接并行传输）、header压缩、server push
   3.0：QUIC协议（UDP），解决队头阻塞

5. Cookie vs Session：
   Cookie：浏览器存储，key-value，可设置过期时间
   Session：服务端存储，依赖Cookie中的sessionId
```

#### TCP/IP ⭐⭐⭐

```markdown
1. 三次握手：
   C -> S：SYN=1, seq=x（客户端发连接请求）
   S -> C：SYN=1, ACK=1, seq=y, ack=x+1（服务端同意）
   C -> S：ACK=1, seq=x+1, ack=y+1（客户端确认）
   → 为什么三次：防止历史连接初始化、双方确认收发能力

2. 四次挥手：
   C -> S：FIN=1（客户端请求关闭）
   S -> C：ACK=1（服务端确认，但可能还有数据）
   S -> C：FIN=1（服务端发送完数据，请求关闭）
   C -> S：ACK=1（客户端确认关闭）
   → TIME_WAIT：2MSL（Linux默认60秒），确保对方收到最后ACK

3. TCP vs UDP：
   - TCP：面向连接、可靠、有序、流量控制（滑动窗口）、拥塞控制
   - UDP：无连接、不可靠、无序、无流量/拥塞控制
   → DNS、实时音视频用UDP；HTTP/HTTPS、SSH、邮件用TCP

4. 滑动窗口与流量控制：
   接收方窗口（rwnd）告诉发送方最多发多少数据
   拥塞窗口（cwnd）是发送方根据网络状况自己调整

5. 粘包/拆包：
   原因：TCP是字节流协议，无消息边界
   解决：固定长度、特殊分隔符、消息头+消息体（Length字段）

6. HTTP与TCP的关系：
   HTTP/1.1：短连接（默认）→ 长连接（Connection: keep-alive）
   HTTP/2：多路复用（复用TCP连接）
   HTTP/3：QUIC（UDP）替代TCP，解决队头阻塞
```

#### HTTPS ⭐⭐

```markdown
1. TLS握手流程：
   ① 客户端问候：支持的加密套件、TLS版本、随机数client_random
   ② 服务端问候：选定的加密套件、证书、随机数server_random
   ③ 客户端验证证书（CA机构验证）
   ④ 密钥交换：用服务器公钥加密预主密钥
   ⑤ 双方用预主密钥+两个随机数生成对称密钥
   ⑥ 之后用对称密钥加密通信

2. 对称加密 vs 非对称加密：
   - 对称加密：AES，速度快，用于实际数据传输
   - 非对称加密：RSA，密钥交换和身份认证
   → HTTPS两者结合：握手用非对称加密，之后用对称加密

3. 证书链验证：
   根CA → 中间CA → 服务器证书
   浏览器内置根CA公钥，验证证书链

4. 中间人攻击：
   HTTPS能防止中间人攻击，因为攻击者没有服务器私钥，无法伪造有效证书
```

### 4.4 操作系统高频

#### 进程与线程 ⭐⭐⭐

```markdown
1. 进程 vs 线程：
   进程：资源分配单位（内存、文件句柄），独立地址空间，进程间通信需IPC
   线程：CPU调度单位，同一进程内共享内存和资源，上下文切换开销小
   → 进程是容器，线程是执行单元

2. 进程状态：
   创建→就绪→运行→阻塞→终止
   阻塞：等待I/O、信号量等事件

3. 进程间通信（IPC）：
   - 管道（pipe，匿名管道用于父子进程）
   - 命名管道（FIFO）
   - 消息队列
   - 共享内存（最快，需要同步）
   - 信号量（PV操作，进程同步）
   - Socket（跨主机通信）
   - 信号（软中断通知）

4. 线程同步方式：
   - 互斥锁（mutex）
   - 读写锁（多读单写）
   - 条件变量（wait/notify）
   - 信号量（P/V操作）
   - 屏障（barrier）

5. 调度算法：
   - FCFS：简单，但可能产生长作业等待（convoy效应）
   - SJF：短作业优先，理论上最优，但难以预知执行时间
   - RR（时间片轮转）：公平，适用于分时系统
   - 优先级调度：可能导致低优先级饥饿（aging解决）
```

#### 内存管理 ⭐⭐⭐

```markdown
1. 虚拟内存：
   - 进程使用虚拟地址，通过MMU映射到物理地址
   - 页面置换算法：LRU、LFU、FIFO、Clock算法
   → 让程序使用比物理内存更大的地址空间

2. 分页 vs 分段：
   - 分页：固定大小页（如4KB），消除外部碎片，但可能内部碎片
   - 分段：按逻辑含义分段（代码段、数据段），方便共享和权限控制
   - 段页式：先分段再分页，结合两者优点

3. 页面置换算法：
   - OPT（理想）：置换最长时间不用的（不可实现，用于评估）
   - FIFO：简单，但Belady异常（更多帧反而更多缺页）
   - LRU：置换最久未使用的（近似实现成本高）
   - Clock：环形链表+引用位，LRU近似实现

4. 内存碎片：
   - 外部碎片：内存块之间的小空洞（分段导致）
   - 内部碎片：分配块内未被使用部分（分页导致）
   → 伙伴系统解决外部碎片

5. 堆 vs 栈：
   - 栈：自动分配释放，大小固定，函数调用帧，生长方向向下
   - 堆：手动申请释放（malloc/free、new/delete），动态分配，向上生长
```

#### IO模型 ⭐⭐

```markdown
五种IO模型：

1. 阻塞IO（BIO）：进程阻塞等待数据准备好
   → 简单，但效率低，一个连接一个线程

2. 非阻塞IO（NIO）：轮询检查数据是否就绪
   → 仍需不断轮询，CPU空转浪费

3. IO多路复用（select/poll/epoll）：
   - select：fd数组，默认1024，线性扫描（O(n)）
   - poll：链表，无上限，但仍是线性扫描
   - epoll：红黑树+链表，事件驱动，只返回就绪的（O(1)）
   → Linux高性能网络编程核心

4. 信号驱动IO：数据就绪时发信号通知
   → 不常用

5. 异步IO（AIO）：全程不阻塞，数据复制完成后通知
   → Linux 7+才成熟，Windows IOCP是成熟实现

epoll三个关键函数：
- epoll_create：创建实例
- epoll_ctl：注册/删除/修改监听事件
- epoll_wait：阻塞等待事件就绪
```

### 4.5 设计模式与算法思想

#### 常见设计模式

```markdown
单例模式 ⭐⭐⭐
- 饿汉式：类加载时创建，无线程安全问题，但浪费资源
- 懒汉式：使用时创建，需要双重检查锁定（DCL）
- 枚举：最简洁，天然防反射和反序列化破坏

工厂模式 ⭐⭐
- 简单工厂：factory根据参数创建对象
- 工厂方法：子类决定实例化哪个类（延迟到子类）
- 抽象工厂：创建一系列相关对象

代理模式 ⭐⭐
- 静态代理：编译时生成代理类
- 动态代理：JDK Proxy（基于接口）或CGLIB（基于继承）
- 应用：Spring AOP、MyBatis懒加载、RPC框架

策略模式 ⭐⭐
- 定义一系列算法，封装起来可互换
- 应用：支付方式选择、排序算法选择

观察者模式 ⭐
- 一对多依赖，主题状态变化自动通知观察者
- 应用：事件监听、消息订阅、Spring事件机制

装饰器模式 ⭐
- 动态给对象加功能，比继承更灵活
- IO流体系：BufferedInputStream包装FileInputStream

适配器模式 ⭐
- 将不兼容接口转换为可兼容接口
- 应用：日志框架adapter、第三方SDK集成
```

#### 算法思想

```markdown
分治法 ⭐⭐
- 思想：将大问题拆成子问题，解决后合并
- 典型：归并排序、快速排序、二分查找、归并排序求逆序对

动态规划 ⭐⭐⭐
- 思想：最优子结构 + 重叠子问题
- 解法：自底向上填表 或 自顶向下记忆化递归
- 经典问题：爬楼梯、斐波那契、背包问题、最长公共子序列

回溯法 ⭐⭐
- 思想：穷举搜索 + 剪枝
- 应用：全排列、组合、8皇后、数独
- 模板：选择 → 递归 → 撤销选择

贪心算法 ⭐
- 思想：每步取最优（局部最优），期望全局最优
- 关键：证明贪心选择性和最优子结构
- 经典：活动选择、区间调度、Huffman编码

双指针 ⭐⭐⭐
- 同向双指针：滑动窗口（数组/字符串问题）
- 相向双指针：两数之和、有序数组平方
- 快慢指针：链表环检测、删除倒数第N个节点

二分查找 ⭐⭐⭐
- 前提：有序数组
- 变体：查找左边界、查找右边界、旋转数组搜索
- 常见坑：整型溢出（用mid = left + (right-left)/2）

前缀和 ⭐⭐
- 一维前缀和：快速求区间和
- 二维前缀和：快速求矩阵区域和
- 差分数组：快速区间增减 + O(1)区间查询
```
