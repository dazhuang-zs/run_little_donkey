# 02 求职准备与简历——普通院校计算机生存手册

> 本文面向双非/二本计算机专业学生，提供从时间规划到笔试准备的完整求职路线图。信息密集，可直接照做。

---

## 第一部分：求职时间线规划

### 1.1 秋招/春招时间线（7月—次年5月关键节点）

国内互联网/大厂校招分两季，节奏固定，提前布局是普通院校学生逆袭的核心武器。

#### 秋招时间线（大三暑假—大四上）

| 时间 | 阶段 | 关键动作 |
|------|------|----------|
| **7月上旬** | 提前批开放 | 字节、腾讯、百度、美团等大厂陆续开放提前批，不占正式批名额，无损尝试 |
| **7月中旬—8月** | 大厂提前批密集期 | 阿里、华为、网易提前批；小厂/外企同步开放；**这是普通院校学生最重要的窗口** |
| **9月1日** | 秋招正式启动 | 各大公司同步开放校招官网，每天跟进新开职位 |
| **9月—10月** | 秋招黄金期 | 投递量最大、岗位最多；一线大厂 OC（Offer Call）多集中在10月底 |
| **11月** | 秋招收尾 | 部分中小厂仍在招聘；已拿 Offer 的同学可以谨慎比较 |
| **12月—次年1月** | 补录/冬招 | 大厂 HC 未满时补录；字节、快手等全年滚动招聘 |

**提前批重要性**：提前批通过率通常高于正式批（竞争压力小 30%—50%），且部分公司会将提前批表现纳入正式批参考。**普通院校学生必须把提前批作为主攻方向**，不要等到9月再动手。

#### 春招时间线（次年3月—5月）

| 时间 | 阶段 | 关键动作 |
|------|------|----------|
| **3月初** | 春招启动 | 以中小厂、外企为主；大厂 HC 较少，主要补招未完成的岗位 |
| **3月—4月** | 春招主力期 | 腾讯、阿里、字节均有春招名额，但总量约为秋招的 20%—30% |
| **5月** | 春招收尾 | 部分公司招聘结束；毕业前最后窗口 |

**春招 vs 秋招**：秋招是主战场，岗位数量是春招的 3—5 倍。秋招失利才打春招，但备战策略不变。

### 1.2 实习 vs 正式岗的区别与策略

#### 实习的核心价值

1. **简历加分**：大厂实习经历等价于 985 学历的部分权重，大三有大厂实习 ≈ 秋招通过率提升 40%
2. **转正通道**：字节、腾讯、阿里均有实习转正机制，转正 Offer 竞争压力远低于社会招聘
3. **提前探路**：了解真实工作内容，避免入职后方向不匹配

#### 实习时间策略

- **大二下—大三上**：冲小厂/中厂实习，积累第一段经历
- **大三下（寒假/春季）**：冲大厂日常实习（提前一学期布局）
- **大三暑假**：暑期实习是最重要的节点，字节/腾讯/阿里暑期实习直接关联秋招转正

#### 正式岗注意事项

- 正式校招要求**应届生身份**（毕业当年及毕业前一年），注意保护应届身份
- 不要因为实习"转正"就忽略秋招投递，双线并行才安全
- 部分公司（如华为、中兴）技术岗要求全程参与校招流程，无法直接转正

---

### 1.3 普通院校学生的投递节奏建议

**核心原则：以量保质，精准筛选，早动手**

#### 投递量参考

| 院校层次 | 建议投递量（秋招） | 预期面试量 |
|----------|------------------|------------|
| 985/211 | 15—30 家 | 10—20 家 |
| **双非/二本** | **50—100 家** | **15—25 家** |

普通院校简历通过率约为 985 的 40%—60%，需要以更大投递量覆盖。

#### 分层投递策略

将目标公司分为三层，按 **3:5:2** 比例分配精力：

- **冲刺层（30%）**：一线大厂（BAT、字节、美团、滴滴），结果看天，重在练手
- **核心层（50%）**：二三线大厂（京东、小米、哔哩哔哩、快手、网易游戏、各行业独角兽），主要目标
- **保底层（20%）**：地方国企、银行科技部、外包大厂（IBM、埃森哲、软通动力），托底用

#### 投递节奏

- **提前批（7—8月）**：集中冲刺冲刺层，失败不影响正式批
- **9月—10月**：每天上午投简历（招聘官早上处理简历），下午准备面试
- **保持弹药**：不要一次性全部投出，留 30% 在 10 月中下旬投递，覆盖晚开的岗位

## 第二部分：简历撰写指南

### 2.1 技术简历的核心结构

一份合格的技术简历控制在 **1 页**（应届生），信息密度高于格式美观。

#### 模块顺序（从上到下）

```
姓名 | 手机号 | 邮箱 | GitHub | 技术博客（可选）
—————————————————————————————————————————
教育背景
技术栈
项目经历（2—3 个，核心）
实习经历（有则写，无则省略）
竞赛/开源贡献（加分项）
```

#### 各模块要点

**① 个人信息**

- 必填：姓名、手机、邮箱、GitHub（务必有内容！）
- 加分：LeetCode 主页（题数 200+ 可写）、技术博客（有质量文章才写）
- 不要写：照片（科技公司不需要）、年龄、籍贯、政治面貌（除非招聘方要求）

**② 教育背景**

格式：`XX大学 | 计算机科学与技术 | 本科 | 2021.09—2025.06 | GPA: 3.5/4.0`

- GPA 高于 3.5/4.0 或排名前 20% 才写，否则省略
- 有相关专业课成绩亮点（如操作系统 95+）可以单独列出
- 双非学生**绝对不要造假学历**，大厂背调 100% 会查

**③ 技术栈**

按掌握程度诚实分级，常用格式：

```
编程语言：Java（熟练）、Go（了解）、Python（了解）
框架/中间件：Spring Boot、MyBatis、Redis、Kafka、MySQL
工具：Git、Docker、Linux、IntelliJ IDEA
```

- **不要堆砌**：写了就要能答出来，面试官会针对每一项提问
- **不要写"精通"**：90% 的应届生写精通是减分项，用"熟练/了解"更可信

**④ 项目经历（最重要的模块）**

详见 2.2 STAR 法则章节。

**⑤ 实习经历**

格式：`公司名 | 岗位 | 时间 | 主要职责（2—3 条，STAR 格式）`

- 无实习经历不要硬凑，留白比造假好
- 小公司实习也要写，体现工程经验

---

### 2.2 项目描述 STAR 法则

STAR = **S**ituation（背景）+ **T**ask（任务）+ **A**ction（行动）+ **R**esult（结果）

大多数同学的项目描述只有 A（做了什么），缺少 S（为什么）和 R（效果如何），导致简历苍白无力。

#### 示例对比

**差的写法**（纯 Action）：
> 使用 Spring Boot + Redis 实现了用户登录功能，支持 Token 验证和权限控制。

**好的写法**（完整 STAR）：
> **S**：原系统用户鉴权依赖数据库查询，高并发下接口响应时间达 500ms+；
> **T**：负责重构鉴权模块，目标将响应时间降低至 100ms 以内；
> **A**：引入 Redis 缓存 Token，设计基于 JWT 的无状态鉴权方案，并通过 Lua 脚本保证 Token 刷新原子性；
> **R**：鉴权接口响应时间降至 **35ms**，QPS 支撑能力从 800 提升至 **5000+**，压测通过。

#### 关键原则

1. **结果必须量化**：响应时间 / QPS / 减少的错误率 / 提升的吞吐量，没有数字的结果不可信
2. **技术选型要有理由**：为什么用 Redis 不用 Memcached？体现你的思考而非堆技术词
3. **体现个人贡献**：团队项目要说清楚"我负责了什么"，不要写成团队介绍
4. **控制长度**：每个项目 3—5 条 bullet，每条 1—2 行，面试官扫简历时间约 30 秒

### 2.3 普通院校学生如何弥补学历短板

学历是门槛，但不是终点。以下三个方向可以有效补足差距：

#### ① 开源贡献

**为什么有效**：开源贡献是可验证的技术实力证明，GitHub PR 记录胜过任何自我描述。

**如何入手**：
- 从使用的框架开始：给 Spring Boot、MyBatis 等提 Issue 或修 Bug
- 寻找 `good first issue` 标签的 PR 任务，门槛低，容易起步
- 中文友好的社区：Apache Dubbo、Seata、ShardingSphere 均有中文维护者
- **目标**：秋招前有 **2—3 个合并的 PR**，写进简历有力度

**简历写法**：
> Apache Seata Contributor：修复了 GlobalTransactional 注解在多数据源场景下失效的 Bug（PR #4521，已合并）

#### ② 技术博客

**为什么有效**：体现学习深度和表达能力，面试官会在面试前搜索候选人博客。

**平台选择**：
- **博客园 / 掘金**：国内技术社区，流量大，容易被搜索到
- **个人网站**（GitHub Pages + Hexo）：加分项，体现工程能力

**内容方向**：
- 源码分析：Redis 跳表实现原理、HashMap 扩容机制
- 踩坑记录：项目中遇到的真实问题及解决过程
- 读书笔记：《深入理解 Java 虚拟机》《MySQL 技术内幕》

**目标**：秋招前 **10—15 篇有深度文章**，避免写"Hello World 教程"类入门文

#### ③ 项目深度

**为什么有效**：大厂面试官看项目不是看"做了什么"，而是看"遇到了什么问题、怎么解决"。

**深度体现方向**：
- **并发问题**：你的项目有没有处理过并发安全？用了什么锁？
- **性能优化**：有没有做过压测？瓶颈在哪？怎么解决的？
- **高可用设计**：有没有考虑过服务挂掉怎么办？有没有降级策略？
- **数据量**：百万级数据怎么分页查询？怎么建索引？

**建议**：做 1—2 个有一定复杂度的个人项目，而非 5—6 个 CRUD 玩具，深度 > 数量。

**推荐项目方向**：
- 简化版消息队列（体现网络编程 + 并发）
- 秒杀系统（体现高并发 + Redis + 分布式锁）
- RPC 框架简化实现（体现网络 + 反射 + 序列化）

### 2.4 常见简历错误与避坑指南

| 错误类型 | 具体表现 | 正确做法 |
|----------|----------|----------|
| **技术词堆砌** | "精通 Java、Spring、Redis、Kafka、Docker、K8s..." | 只写真正会用的，分级标注 |
| **项目无结果** | "实现了用户管理模块" | 加上量化结果："支持 10 万用户并发登录" |
| **模板痕迹** | "具有良好的沟通能力和团队合作精神" | 删掉所有软技能空话，用项目经历说话 |
| **简历超页** | 2—3 页简历 | 应届生强制 1 页，精简到核心 |
| **格式混乱** | Word 排版乱、字体大小不一 | 用 LaTeX 或 Typora 导出 PDF，保证格式稳定 |
| **邮箱不专业** | `loveyou123@qq.com` | 换成 `姓名拼音@gmail.com` 或 `@163.com` |
| **GitHub 空仓库** | 写了 GitHub 链接但仓库没有代码 | 要么填充内容，要么删掉链接 |
| **时间线错误** | 实习时间和学校时间冲突 | 背调会查，务必真实 |
| **缺乏针对性** | 同一份简历投所有公司 | 针对后端/前端/算法岗微调技术栈顺序 |

## 第三部分：投递渠道与策略

### 3.1 主流渠道对比

| 渠道 | 类型 | 优势 | 局限 | 适用场景 |
|------|------|------|------|----------|
| **各公司校招官网** | 官方渠道 | 信息权威，流程完整 | 需逐一注册，效率低 | 所有目标公司必投 |
| **牛客网** | 聚合平台 | 校招信息聚合，有笔试题库 | 部分岗位非官方 | 信息收集 + 笔试练习 |
| **Boss 直聘** | 社会招聘为主 | 可直接联系 HR/技术 Lead | 校招岗位少，多为社招 | 日常实习、春招补充 |
| **内推** | 人脉渠道 | 简历直达，通过率提升明显 | 需要人脉积累 | 核心目标公司必争取 |
| **实习僧** | 实习专项 | 小厂实习岗位多 | 质量参差不齐 | 找第一段实习经历 |
| **LinkedIn** | 外企为主 | 外企/跨国公司 HR 活跃 | 国内大厂少 | 冲击外企岗位 |

**使用建议**：
1. **校招官网**是基础，目标公司 100% 通过官网投递
2. **牛客**订阅校招信息推送，不要每天手动找
3. **Boss 直聘**用于实习和春招期间联系中小公司
4. **内推**是最高优先级，获取方式见 3.3

---

### 3.2 海投 vs 精投的选择

这不是非此即彼的选择，而是**分阶段组合使用**。

#### 什么时候海投

- **提前批阶段（7—8月）**：提前批竞争压力小，海投大厂有意义，失败无损耗
- **秋招正式批（9—10月）**：目标公司 50+ 家时，无法逐一深度研究，需要一定海投覆盖率
- **保底阶段**：当已投公司面试进展缓慢，补充投递中小厂托底

#### 什么时候精投

- **有内推资源时**：内推资源有限，必须精准用在核心目标公司
- **已有面试机会时**：同期 3—5 家面试已经精力饱和，暂停新投递，集中备战
- **针对某类公司**：如专攻游戏行业，需要深研目标公司技术栈和产品

#### 实操建议

- 维护一张投递表（Excel/飞书多维表格），记录公司名、投递时间、状态、跟进节点
- 每天固定 1—2 小时投简历，不要一次性投完就不管
- 简历投出后 **7 天无回音**，可通过牛客/脉脉查询该公司 HR 联系方式催进度

---

### 3.3 内推的价值与获取方式

#### 内推的真实效果

- 简历直达用人部门，绕过 HR 初筛池，**通过率提升 30%—60%**
- 部分公司内推有专属流程，笔试可以跳过或优先安排
- 内推人可以跟踪进度，面试失败时可以获得反馈

#### 内推的获取方式（从易到难）

**① 师兄师姐网络**

- 加入学校 CS 学生群、校友群，秋招季主动询问"有没有在 XX 公司的学长能内推"
- 学校就业办通常维护校友信息库，可以申请联系

**② 牛客内推板块**

- 牛客"内推"频道每年秋招有大量员工公开发布内推码
- 使用内推码投递时，务必附上一封简短的自我介绍邮件给内推人

**③ GitHub / 技术博客**

- 关注目标公司技术博客或开源项目的 Contributor，发邮件请求内推
- 前提：你的 GitHub 或博客有足够内容，让对方愿意推你

**④ LinkedIn / 脉脉**

- 直接搜索目标公司 + 岗位，联系同岗同学/员工
- 信息模板：简短、具体、说明自己的背景和诉求，不要发模板群发

**⑤ 技术社区建立关系**

- 在掘金、思否等社区与目标公司员工互动，建立弱连接
- 参加线下技术沙龙/Hackathon，直接认识工程师

**注意事项**：
- 内推前先完善好简历，不要仓促内推后简历出问题
- 内推成功后正常走面试流程，不要因为有内推就放松备战
- 感谢内推人，无论结果如何，维护好关系

## 第四部分：笔试准备

### 4.1 技术笔试类型

国内互联网校招笔试主要分三类，不同公司侧重不同：

| 类型 | 代表公司 | 内容 | 准备重点 |
|------|----------|------|----------|
| **算法题** | 字节跳动、百度、微软 | 2—4 道编程题，OJ 在线评测 | LeetCode 刷题，见 4.2 |
| **八股文** | 阿里、腾讯、美团 | 选择题 + 简答，考 Java/计网/OS | 知识点整理，见 4.3 |
| **场景设计** | 阿里、蚂蚁、网易 | 系统设计题（如设计一个短链服务） | 理解高可用设计模式 |

大多数公司同时考算法 + 八股，场景设计出现在笔试后的面试环节较多。

---

### 4.2 算法题准备策略（LeetCode 刷题路线）

#### 总体目标

- **保底**：LeetCode 前 150 题（LeetCode 150 精选），可以通过大部分中厂笔试
- **目标**：300—500 题，覆盖主流题型，通过大厂笔试
- **冲刺**：500+ 题 + 竞赛题，字节/微软高难度笔试

时间规划：如果秋招 9 月开始，**建议从大三下学期（3月）开始刷**，保证 6 个月准备时间。

#### 分阶段路线

**阶段一：基础打通（4—6 周）**

专注数据结构基础，每种题型做 10—20 题：
- 数组与字符串（双指针、滑动窗口）
- 链表（反转、合并、快慢指针）
- 栈与队列（单调栈、BFS）
- 哈希表（Two Sum 系列）
- 二叉树（DFS/BFS 遍历）

**阶段二：核心算法（6—8 周）**

- 二分查找（经典模板 + 变种）
- 动态规划（背包、最长子序列、区间 DP）
- 回溯（全排列、子集、N 皇后）
- 图论（DFS/BFS、拓扑排序、最短路径）
- 排序（手写快排、归并排序）

**阶段三：提速与模拟（持续进行）**

- 每周 2—3 次限时模拟（45 分钟内完成 2 题中等难度）
- 参加 LeetCode 周赛，适应竞赛节奏
- 复盘错题，整理解题思路模板

#### 刷题工具与资源

- **LeetCode 精选题单**：Hot 100、LeetCode 150、剑指 Offer
- **《算法导论》**：理论基础，选读（第 2、6、7、15—16 章）
- **labuladong 的算法小抄**：中文题解，框架化解题思路
- **代码随想录**：国内最全系统刷题路线，免费

#### 笔试实战技巧

1. **先过样例，再提交**：调试时间宝贵，不要盲目提交
2. **部分分策略**：算法题答不出最优解，先写暴力解拿部分分，再优化
3. **注意输入输出格式**：很多人因为 I/O 格式错误导致 0 分
4. **选语言**：Java/Python 均可，Python 语法简洁但有时超时；Java 需注意 `Scanner` vs `BufferedReader` 的速度差异

### 4.3 八股文整理方法

八股文是有规律可循的，复习核心在于**理解框架、举一反三、自己输出**。

#### 核心考察范围

**Java 基础**
- 集合框架：HashMap 底层实现（数组+链表+红黑树）、扩容机制、线程安全问题
- 并发：synchronized vs Lock、线程池参数（7 个参数）、AQS 原理、volatile 内存模型
- JVM：内存区域划分、GC 算法（标记清除/整理/复制）、类加载机制

**数据库（MySQL 为主）**
- 索引：B+ 树结构、聚簇索引 vs 非聚簇索引、索引失效场景
- 事务：ACID、隔离级别（四种）、MVCC 实现原理
- 锁：行锁 vs 表锁、间隙锁、死锁检测
- SQL 优化：EXPLAIN 分析、慢查询优化思路

**计算机网络**
- TCP：三次握手/四次挥手（画图！）、TCP 可靠性机制（超时重传、滑动窗口、拥塞控制）
- HTTP：HTTP vs HTTPS、HTTP/1.1 vs HTTP/2、常见状态码
- 其他：DNS 解析过程、CDN 原理、Cookie vs Session

**操作系统**
- 进程 vs 线程 vs 协程
- 内存管理：虚拟内存、页表、TLB
- IO 模型：同步/异步/阻塞/非阻塞、select/poll/epoll 区别
- 锁与同步：互斥锁、读写锁、信号量

**Redis**
- 数据结构：String/List/Hash/Set/ZSet 底层实现
- 持久化：RDB vs AOF
- 常见问题：缓存穿透/击穿/雪崩及解决方案
- 分布式：Redis Cluster 原理、主从复制

#### 整理方法

**① 不要背答案，要画框架图**

以"HashMap 底层原理"为例，先画出：
```
数组（Node[] table）
  └── 链表（节点 < 8）
  └── 红黑树（节点 ≥ 8）
扩容：负载因子 0.75，容量翻倍，重新哈希
线程不安全：并发 put 可能死循环（JDK7）/ 数据丢失（JDK8）
```

**② 自己出题，自己答**

将每个知识点转化为可能被面试官问的问题，然后用自己的话回答，录音回听，修正表达。

**③ 用 Anki 做记忆卡片**

核心结论性知识（TCP 三次握手步骤、GC 算法名称）用 Anki 做闪卡，利用碎片时间复习。

**④ 知识点来源推荐**

- **JavaGuide**（github.com/Snailclimb/JavaGuide）：国内最全 Java 八股文合集
- **小林 coding**（xiaolincoding.com）：图解计算机网络 + 操作系统，易理解
- **《MySQL 是怎样运行的》**：国内 MySQL 最易读的深度书
- **《深入理解 Java 虚拟机》第 3 版**：JVM 必读

#### 答题技巧

- **先说结论，再展开细节**：面试官时间有限，不要绕弯子
- **主动引导深度**："关于 HashMap 我还了解了 ConcurrentHashMap 的分段锁设计，需要展开吗？"
- **不懂就说不懂**：比乱猜一通更好，展示你的边界意识
