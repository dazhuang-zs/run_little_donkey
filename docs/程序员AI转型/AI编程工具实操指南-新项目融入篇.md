# AI 编程工具实操指南：新项目快速融入篇

> 适用对象：第一次接触 AI 编程工具，不知道怎么用、用来干嘛的人。
> 读完时间：约 20 分钟
> 读完能：接手任何一个陌生项目，30 分钟内搞清楚它在做什么、怎么做

---

## 一、什么是 AI 编程工具

AI 编程工具是"会读代码、会写代码、会上网查资料"的 AI 助手，但它能直接操作你的代码文件。

常见的四类工具：

| 工具 | 特点 | 适合谁 |
|------|------|--------|
| **Claude Code** | 最强推理能力，适合复杂任务 | 想深度理解项目的人 |
| **Cursor** | 界面友好，Tab 键自动补全 | 写代码时实时辅助 |
| **GitHub Copilot** | 嵌入 IDE，悄无声息补全 | 不想改变习惯的人 |
| **通义灵码** | 中文界面，免费 | 国内用户 |

**本指南以 Claude Code 为例**，其他工具触类旁通。

## 二、第一件事：让 AI 读懂项目结构

拿到一个新项目，不要从第一行代码开始看。直接让 AI 给你一张"地图"。

### 2.1 问 AI：项目是做什么的

在 Claude Code 里输入（或者直接问我也行）：

```
这个项目是做什么的？给我一个 3 句话的项目概述。
```

### 2.2 问 AI：目录结构是什么意思

```
帮我梳理这个项目的目录结构，解释每个目录的作用。
重点关注：入口文件在哪、核心业务逻辑在哪、配置文件在哪。
```

AI 会告诉你类似这样的内容：

```
├── backend/          后端服务（API 接口）
│   ├── app/          主应用代码
│   │   ├── api/      路由定义
│   │   ├── models/   数据模型
│   │   └── services/ 业务逻辑
├── frontend/         前端页面
├── docs/             文档
└── requirements.txt 依赖列表
```

### 2.3 问 AI：数据是怎么流动的

```
帮我画出这个项目的数据流向：
用户请求 → 经过了哪些模块 → 最终返回什么
用文字描述，不用画图工具。
```

## 三、接手具体任务的标准流程

假设你被分配了一个新任务，比如："修一个评论功能不显示的 bug"，或者"新增一个导出 CSV 的功能"。

### 流程一：理解现有代码

```
我需要修改这个功能：XXX
请帮我找到相关的代码文件，并解释：
1. 这个功能现在的逻辑是什么
2. 改动它可能影响哪些地方
```

### 流程二：让 AI 帮你写代码

```
基于以上理解，帮我实现：
[描述你要的功能]
要求：
- 风格与现有代码保持一致
- 不要改变现有的接口签名
```

### 流程三：让 AI 帮你检查

```
写完之后，帮我检查这段代码有没有问题：
1. 逻辑错误
2. 潜在的性能问题
3. 安全风险
```

## 四、接手陌生项目的 30 分钟快速上手法

### 第 1-5 分钟：了解全貌

在项目根目录问 AI：

```
我刚接手这个项目，请给我：
1. 一句话描述这个项目在做什么
2. 这个项目的技术栈（用什么语言、框架、数据库）
3. 最重要的 3 个文件是什么（代码量最大或最核心）
```

### 第 6-15 分钟：找到入口

```
帮我找到：
1. 程序的入口文件（main.py / index.js / App.tsx 等）
2. 路由/页面定义文件
3. 数据库模型定义文件
```

### 第 16-25 分钟：理解核心业务流程

```
用我听得懂的方式，解释一下：
这个项目是怎么处理一个用户请求的？
从头到尾走一遍，用具体文件名，不要抽象描述。
```

### 第 26-30 分钟：找到你负责模块的位置

```
我在找 [模块名] 相关的代码，它在哪个文件里？
帮我在 3 个关键词内定位。
```

## 五、常用指令清单（直接复制用）

### 5.1 理解代码

```
# 解释这个文件在做什么
解释一下 backend/app/api/v1/trip.py 这个文件的逻辑

# 找相关代码
我需要找和 [功能名] 相关的代码，帮我搜一下

# 画数据流
画出用户提交一个请求到收到响应的完整数据流
```

### 5.2 写代码

```
# 帮我写一个新功能
基于现有的 [文件/模块]，帮我新增 [功能描述]
风格要求：[具体要求]

# 翻译代码风格
把这段代码的风格改成和项目其他代码一致的格式

# 补全测试
帮我为 [函数/模块] 写单元测试
```

### 5.3 修 Bug

```
# 帮我分析这个错误
报错信息：[粘贴错误日志]
帮我分析可能的原因

# 帮我找 Bug
这个功能 [描述问题]，我怀疑是 [猜测原因]
帮我确认并修复
```

### 5.4 重构

```
# 帮我理解这段代码
帮我解释 [代码块] 的作用

# 优化代码
这个函数太长了，帮我拆分成更小的函数，保持功能不变

# 移除重复代码
帮我找出这个文件里重复的代码片段，并给出合并建议
```

## 六、重要的使用原则

### 原则一：永远让 AI 说出它在读哪个文件

问 AI 的时候，尽量带上文件名：
```
❌ "这段代码有问题"          ← AI 不知道你说哪段
✅ "backend/trip.py 第 45 行" ← AI 准确定位
```

### 原则二：先理解，再动手

```
第一次问 AI："这个模块是做什么的？"
第二次问 AI："如果要新增 X 功能，从哪里入手？"
第三次才问："帮我写这段新增代码"
```

### 原则三：让 AI 解释变化

每次 AI 写完代码后问：

```
你改了什么？为什么这样改？这样做有什么风险？
```

### 原则四：不懂就追问

```
"这个概念我没听懂，能用更简单的方式解释吗？"
"能给我一个具体的例子吗？"
"这和我刚才说的有什么区别？"
```

### 原则五：定期让 AI 帮你总结

一个功能写完后，让 AI 给你一段总结：

```
用一段话总结我们刚才做了什么，包括：
- 改动了哪些文件
- 解决了什么问题
- 接下来可能还需要做什么
```

## 七、不同场景的 AI 使用策略

| 场景 | 问法 | 注意 |
|------|------|------|
| 接手新项目 | "项目是做什么的 + 目录结构" | 先看结构再看代码 |
| 修 Bug | "报错信息 + 帮我分析" | 同时给 AI 上下文 |
| 写新功能 | "基于现有 X 模块，新增 Y 功能" | 带上文件名 |
| 读别人代码 | "这个文件做什么 + 数据流" | 让 AI 给你讲故事 |
| 学新技术 | "用简单的语言解释 X 概念" | 让 AI 打比方 |
| 优化性能 | "这段代码哪里慢 + 怎么改" | 给 AI 具体的性能数据 |

## 八、常见误区

### ❌ 误区一：直接把需求扔给 AI

```
❌ "帮我做一个用户登录功能"
```

```
✅ "项目中现有的认证方式是什么？
    我需要新增一个用户登录功能，
    应该参照哪个文件的写法？"
```

### ❌ 误区二：让 AI 一次性写完整的大功能

AI 擅长做小块、具体的事。一口气写 500 行容易出错。

```
✅ 拆分成多个小步骤：
"先帮我写数据库模型"
"然后写 API 路由"
"最后写前端调用"
```

### ❌ 误区三：不让 AI 知道上下文

每次提问都假设 AI 完全不了解你的项目。

```
✅ 每隔几个问题就提醒 AI：
"还记得吗，我们在做的是一个行程规划助手"
```

### ❌ 误区四：AI 说什么就信什么

AI 会"一本正经胡说八道"，尤其是：
- 不知道的 API 用法，会编造出来
- 不知道的包名，会凭空捏造

**验证方法**：
```
这个说法对吗？去查一下官方文档确认。
```

## 十、快速开始命令

如果你是第一次用 Claude Code，在项目目录下运行：

```bash
# 进入项目
cd your-project-path

# 启动 Claude Code
claude

# 第一个问题
项目中最重要的 3 个文件是什么？给我一句话说明每个文件的作用。
```

**记住：AI 是你的搭档，不是你的替代品。**

它能帮你读懂代码、写出代码，但它不懂你的业务、你的用户。

**你负责方向，AI 负责执行。**
